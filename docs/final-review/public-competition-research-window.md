# 满帮连续找货智能体比赛复盘：从司机偏好理解到在线决策

这次比赛要求参赛者设计一个货运司机 Agent。Agent 需要在官方仿真环境中持续观察司机状态和当前可见货源，并在 `take_order`、`wait`、`reposition` 三类动作中作出选择，目标是在控制运输成本和偏好罚分的同时，提高整个仿真周期的净收益。

与常见的预测类比赛不同，这道题没有固定的单次输入输出。每次接单都会改变司机的位置、时间和后续可见货源，等待和空驶也会影响之后的选择。因此，我们把它理解为一个带自然语言偏好约束的在线序贯决策问题。

## 一、这道题主要难在哪里

Agent 面对的不只是“哪票货价格更高”，还要同时处理三类问题：

1. **货源收益与成本。** 需要综合货价、赴装空驶、载货距离、运输时间和等待时间，而不是只比较订单价格。
2. **连续决策。** 当前动作会改变下一步的时间、位置和候选货源，单步最优不一定等于整个周期最优。
3. **司机自然语言偏好。** 司机可能要求夜间休息、完成月度货源配额、限制长途订单数量、避开特定地区，或者在指定时间到达某个城市。

其中最难的是第三类。偏好由自然语言描述，既有时间、地点和数量条件，也可能包含转折、例外和跨周期关系。例如：

- 每天 21:00 至次日 06:00 必须停车休息，但周末可以晚两小时开始。
- 四月某类货必须接满 12 单，未完成的数量需要在五月继续补。
- 每月运输时间超过 8 小时的长途订单最多接 5 单。
- 某个月需要到指定城市停留若干天。

如果直接为每句话编写一套 `if-else`，规则数量会迅速增加，复合偏好也很难正确组合。我们的主要思路，是先把偏好编译成统一的结构，再交给后续规划模块执行。

## 二、整体方案

系统的主要链路如下：

```text
司机状态与当前可见货源
          |
          +--> 自然语言偏好 -> Preference Compiler -> 可执行规则与任务
          |                                             |
          +------------------> 候选货源评分 <-----------+
                                      |
                                      v
                              ServicePlan 在线规划
                                      |
                                      v
                       take_order / wait / reposition
```

其中，偏好处理部分采用了更细的编译链路：

```text
自然语言偏好
  -> Preference IR
  -> Rule DSL / Task Template
  -> Driver Progress Ledger
  -> Runtime Scoring
  -> ServicePlan
```

`Preference IR` 是偏好的中间表示，用统一字段保存时间范围、地点、数量、周期、适用对象和罚分等语义；`Rule DSL` 和 `Task Template` 则把这些语义转换成运行时能够执行的规则或任务。这样，LLM 负责理解语言，Python 模块负责保证结构和执行逻辑稳定。

## 三、把偏好归纳为通用类型

为了适配隐藏司机，我们没有针对某个司机、城市、品类或月份编写专属逻辑，而是把偏好归纳为几类通用 effect family：

| 类型 | 用途 | 示例 |
| --- | --- | --- |
| `time_window_task` | 处理休息、禁行和特定时间窗任务 | 每天夜间停车休息 |
| `location_task` | 处理到达某地、在某地停留等要求 | 本月到某城市停留若干天 |
| `quantity_goal` | 处理周期内最低完成数量 | 本月某类货至少接 12 单 |
| `carryover_goal` | 处理上期欠额转入下期 | 四月欠额需要在五月补齐 |
| `quantity_limit` | 处理周期内数量上限 | 每月长途订单最多 5 单 |
| `avoidance_rule` | 处理货源、地区或时间规避 | 不接装货地在某城市的订单 |
| `score_rule` | 对满足条件的候选货源加分或扣分 | 空驶超过阈值时降低候选价值 |
| `advisory` | 保存暂时缺少必要字段、不能安全执行的偏好 | 只有倾向但没有明确条件的描述 |

这层抽象的作用，是让不同表述先进入有限的规则类型，再由统一运行时处理，而不是不断增加新的特殊类别。

## 四、Preference Compiler 如何工作

偏好编译器由本地解析、LLM 语义抽取和确定性校验共同组成。

### 1. 本地解析与 LLM 抽取

本地解析器负责识别较明确的时间、月份、数量、地点和比较关系。对于包含转折、例外、跨期补偿或多重条件的句子，再调用 LLM 抽取结构化语义。

LLM 不直接输出最终动作，也不直接修改策略权重。它只输出受 schema 限制的候选 `Preference IR`，之后仍要经过 Python 侧校验。

### 2. canonicalizer

`canonicalizer` 是字段规范化模块，负责把本地解析和 LLM 输出统一为稳定格式，例如：

- 将中文月份、相对月份转换成统一周期字段。
- 将“至少”“最多”“少一单扣一次”等表达转换成明确的比较关系和计数方式。
- 统一跨日时间窗、地点名称、货源属性和罚分字段。
- 为规则生成稳定的 rule key、ledger key 和 runtime metric。

它解决的是“LLM 理解大致正确，但字段形式不稳定”的问题。

### 3. coverage gate

`coverage gate` 用于检查原文中的关键条件是否都进入了 IR。例如一句话同时包含“每月”“运输时间超过 8 小时”和“最多 5 单”，那么周期、判定阈值和数量上限都必须存在；缺少任一关键字段，规则都不能直接进入运行时。

### 4. dry-run

`dry-run` 是规则试运行。系统会使用最小的模拟上下文执行规则，检查字段是否完整、比较关系是否有效、ledger 是否可更新，以及 runtime scoring 是否能够正常计算。

### 5. runtime-ready gate

`runtime-ready gate` 是编译链路的最后一道门。只有通过 schema、coverage 和 dry-run 检查的规则，才会被标记为可执行。已经由本地解析器稳定生成的规则，不会被后续 LLM 输出随意覆盖；缺少关键语义时，LLM 主要承担补充和修复作用。

## 五、用 Driver Progress Ledger 记录偏好进度

很多偏好不是一次判断就能完成，而是需要跨多个决策步骤累计状态。因此我们设计了 `Driver Progress Ledger`，即司机进度账本。

账本主要记录：

- 当前周期某类订单已经完成多少单、还差多少单。
- 长途订单已经接了多少单、是否接近上限。
- 上一周期欠额是否已在当前周期补齐。
- 地点任务是否到达、累计停留时间是否满足要求。
- 时间窗任务是否已执行或发生违反。

每次官方历史确认动作完成后，系统再更新账本。规划器使用的是已确认进度，而不是把尚未执行的计划提前计入完成量。

有了这层状态，`quantity_goal` 和 `carryover_goal` 才能根据“剩余缺口”和“剩余时间”动态影响候选评分；`quantity_limit` 也能在接近上限时及时降低相关订单的优先级。

## 六、将偏好任务接入 ServicePlan

运行时决策使用 `ServiceNode` 表示可规划节点。真实货源、等待、偏好时间任务和地点任务都可以转换为 ServiceNode，再由 `ServicePlan` 进行短视界组合。

对于不同类型的偏好，我们采用两种接入方式：

- **任务型偏好：** 休息、到访、停留等要求转换成带时间、地点和价值的虚拟服务节点。
- **货源型偏好：** 品类目标、数量上限、地区规避和空驶阈值等规则，转换成候选货源的加分、扣分或硬约束。

候选方案主要考虑：

- 货源毛收入与运输成本。
- 赴装空驶、载货行驶和等待耗时。
- 单位时间净收益。
- 当前偏好目标缺口和违反风险。
- 完成订单后的地点及后续机会。

我们采用滚动规划方式：每次根据当前可见信息计算一个短计划，只执行第一步；动作完成后重新查询货源、更新账本并再次规划。这样可以适应货源刷新，也不会被较长的旧计划锁住。

## 七、本地与平台结果

项目在公开 demo 上曾得到以下结果：

| 指标 | 结果 |
| --- | ---: |
| 本地公开 demo 最高净收益 | `114856.46` |
| 对应偏好罚分 | `8000.0` |
| 参数实验 runner 本地最好净收益 | `108923.34` |
| 对应偏好罚分 | `3000.0` |
| 平台隐藏集已知最好成绩 | `25108.30` |

平台结果与本地结果差距较大。我们还出现过两次完全相同的 `208300.0` 偏好罚分，这说明问题并不只是货源刷新造成的随机波动，而是某些隐藏偏好被确定性地解析或执行错误。

从结果看，货源搜索、收益评分和短视界规划在公开环境中能够产生较高净收益，但偏好编译没有稳定泛化到隐藏表达。只要约束理解错误，后面的规划器就会在错误目标上继续优化，最终形成较高罚分。

## 八、几个典型问题

### 1. 转折和例外容易被错误扩展

例如“夜间 21:00 至 06:00 休息，周末可以晚两小时开始”，正确含义是周末开始时间变为 23:00，结束时间仍为 06:00。LLM 可能把整个时间窗平移为 23:00 至 08:00。

这类问题要求 IR 明确区分“修改开始时间”和“整体平移时间窗”，不能只保存一个模糊的 `offset_hours`。

### 2. 跨期目标容易混淆

“四月未完成的目标五月补齐，五月另有新的建材指标”同时包含历史欠额和当前目标。若没有分别生成 `carryover_goal` 和 `quantity_goal`，模型可能把两个目标合并，或者把五月指标错误地放到四月。

### 3. 解析正确不等于执行正确

即使 IR 中识别出了“本月接满 12 单”，如果 ledger 没有正确累计进度，或者 ServicePlan 给目标订单的压力不足，最终仍然无法完成任务。因此需要分别验证文本解析、规则生成、状态更新和规划执行，不能只检查 LLM 返回的 JSON。

## 九、这次比赛留下的经验

第一，偏好理解需要按照编译系统来设计。LLM 适合抽取语义，但最终进入规划器的必须是经过规范化、覆盖检查和试运行的确定性规则。

第二，公开数据上的正确率不能直接代表隐藏集泛化。除了原始偏好，还应系统构造同义改写、条件重排、转折、例外、跨期和多约束组合测试。

第三，偏好编译和在线规划要分别验证。编译层回答“规则是什么”，账本层回答“完成了多少”，规划层回答“下一步怎样兼顾收益和规则”。其中任何一层出错，最终分数都会受到影响。

第四，单位时间收益适合在线决策，但不能成为唯一指标。临近截止时间的配额缺口、数量上限和高额罚分需要进入同一个价值函数，否则短期经济性可能挤压长期目标。

## 十、后续可以继续做什么

如果继续完善这个方向，优先级较高的工作有三项：

1. 建立自然语言偏好编译测试集，覆盖同义表达、复合条件、跨期关系、例外和对抗式改写。
2. 为每条规则建立从原文、Preference IR、Rule DSL、ledger 更新到 runtime scoring 的端到端可追踪记录。
3. 在正确约束的基础上研究更稳定的滚动时域优化，使每个动作都能解释对应的收益、成本、偏好贡献和后续状态变化。

这次项目最终形成的核心思路可以概括为：先把司机自然语言偏好稳定编译成可执行约束，再由在线规划器在当前可见信息下持续权衡收益、成本和偏好进度。
