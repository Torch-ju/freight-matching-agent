# 满帮动态决策智能体比赛复盘

## 结论

我们没有赢。已知最好平台结果约为 `25108.3000`，而当时榜首已经达到 `116876.21`。本地完整仿真曾多次跑到 `10.8 万` 到 `11.4 万`，但平台隐藏集出现高额偏好扣分，说明本地收益能力没有稳定泛化到隐藏司机偏好。

核心失败不是“货源价值完全不会算”，而是：

1. 隐藏偏好自然语言没有稳定编译成可执行约束。
2. 本地公开偏好覆盖不足，导致本地罚分低估平台风险。
3. LLM 解析、正则兜底、Python canonicalizer 的权威边界在后期才逐渐清晰。
4. 冲榜后期复杂度上升太快，测试和平台反馈闭环跟不上。

## 我们做对的事

### 1. 明确了在线-only 边界

项目最终形成了比较清晰的竞赛边界：Agent 只能通过官方 API 获取司机状态、货源查询、历史动作和模型调用，不能读取原始 `cargo_dataset.jsonl`、`drivers.json` 或离线全量市场统计。这一点保证了工程方向没有变成“本地偷看数据”。

### 2. 从单票贪心推进到 ServicePlan

后期已经不只是单票 `price - cost` 贪心，而是开始把真实货源、wait 任务、reposition 任务、偏好目标和资源约束统一到 ServiceNode/ServicePlan。这个方向是正确的：比赛本质是有限时间内的在线序贯优化，不是单步分类。

### 3. 偏好工程形成了可复用骨架

`src/preferences/` 里逐渐形成：

- local parser
- LLM parser
- source obligation IR
- rule DSL
- ledger keys
- driver progress ledger
- rule runtime
- task template runtime

虽然最后没有赢，但这个骨架有研究价值。它把“自然语言偏好 -> 中间语义 -> 可执行规则 -> ledger -> 规划器赋值”这条链路拆开了。

### 4. 参数实验 runner 有实际价值

`scripts/run_parameter_experiments.py` 能批量运行本地参数实验、导出 `best_trial.json`、记录毛利、成本、净利、偏好罚分和 token。`trial_006` 证明查询参数会显著影响本地结果。

## 我们做错的事

### 1. 本地公开 demo 高分被过度信任

本地 20260608 demo 主要暴露的是公开司机和公开偏好。我们后期看到 `114856.46` 后，容易认为已经接近榜首水平。但平台隐藏偏好罚分显示，这个本地分数没有足够泛化含义。

正确做法应该是更早建立“偏好编译泛化测试集”，用公开偏好的机制做 paraphrase、组合、反事实和 adversarial 测试，而不是只盯本地完整仿真净利。

### 2. LLM 解析链路权威边界过晚确定

我们曾在 local parser、LLM extractor、verifier、recovery、canonicalizer 之间摇摆。真正应该固定的是：

1. LLM 只负责语义抽取。
2. Python canonicalizer 生成 executable DSL。
3. source-obligation coverage gate 决定是否缺义务。
4. runtime-ready local 输出不能被 LLM 单独覆盖。
5. verifier 只能修 critical missing，不应全量重写。

这些原则后期才明确，已经错过了最佳平台试错窗口。

### 3. 对“同样高罚分”的信号反应太晚

`-133833.78 / 208300` 两次平台结果完全一致，这不是随机性，而是确定性解析错误或兜底规则错误。我们当时已经判断到了，但应更早把工作集中到 trace、IR、coverage gate，而不是继续做参数和执行压力。

### 4. 文档和产物管理失控

项目留下了大量 stage-freeze、旧计划、临时结果、scratch、pytest 临时目录和重复 zip。冲榜阶段可以接受，但赛后必须收敛，否则后续研究者无法判断哪些结论有效。

## 对未来类似比赛的建议

1. 先建“隐藏偏好编译风险测试”，再做高分调参。
2. 任何 LLM parser 都必须有 IR、coverage、dry-run、runtime-ready gate。
3. 平台提交机会少时，每个包只能验证一个假设。
4. 本地仿真净利、偏好罚分、Token、平均耗时要同时看。
5. 每天结束时做一次包和文档清理，避免最后无法复盘。
6. 不要让 LLM 直接决定动作；它更适合低频偏好结构化。
7. 不要把公开司机偏好转成特殊规则；只能抽象成 effect family。

## 可复用结论

如果未来继续研究这个题目，最值得保留的是“自然语言偏好编译器”的方向，而不是后期激进冲榜包本身。真正的问题可以表述为：

> 司机隐藏偏好自然语言如何稳定编译成可执行约束，并与在线运筹决策无缝连接。

这是一个有研究价值的问题，可以独立于比赛继续做。
